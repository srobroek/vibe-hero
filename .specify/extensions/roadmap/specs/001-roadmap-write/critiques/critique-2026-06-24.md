## Executive Summary

The spec and plan are strong: scope is tightly bounded, the determinism split is
expressed cleanly, the constitution check passes on all seven principles, and the design
was already validated by two real dogfood runs. The feature is genuinely de-risked. The
findings below are mostly refinements, not blockers — but three are worth resolving
before `/speckit.tasks` because they would otherwise surface as ambiguity *during*
implementation: (1) the JSON parity contract needs a precise definition of "identical"
(key order, boolean/int encoding), (2) `prd_globs` detection is specified as patterns but
no requirement states what "detection" produces or where it's consumed in *this* spec,
and (3) the spec is silent on how `write` behaves when the roadmap exists but is
malformed/hand-edited. None rise to 🛑.

## Engineering Lens Findings 🔬

### Architecture Soundness
| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E1 | 💡 | `load-config.sh` resolves the repo root by sourcing core `common.sh` with a relative `../../../` walk plus a CWD fallback. This is fragile to where the script is invoked from. | The plan should require load-config to resolve robustly from both the installed path and repo-root path (the tests must cover invocation from a different CWD). |

### Failure Mode Analysis
| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E2 | 🎯 | FR-021/023 require "identical" output across bash/PowerShell, but "identical" is undefined. PowerShell `ConvertTo-Json` and a hand-built bash string can differ in key order, integer vs. quoted-number, boolean casing, and trailing newline — yet be semantically equal. | Define the parity contract precisely: the test compares **normalized** JSON (parsed → canonical key order → re-serialized), OR mandate byte-identical output and pin key order + encoding in both implementations. State which in the plan; the test depends on it. |
| E3 | 💡 | `max_findings` is in this spec's contract but only consumed by the review commands (out of scope here). Carrying it now is fine, but its validation is the only "invalid input" path tested. | Keep it (the contract is shared), but note that `max_findings` is forward-looking so reviewers don't expect it to be *used* in this spec. |

### Security & Privacy
| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E4 | 💡 | YAML is read via grep/sed/awk, not a real parser. A hostile/malformed config could yield surprising values (e.g. injection into the JSON via unescaped content). | Low risk (local dev config), but the bash JSON build MUST escape all interpolated values (it uses `json_escape` — good); add a test fixture with quotes/special chars in a config value to prove escaping. |

### Performance & Scalability
| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E5 | 🤔 | `prd_globs` "detection" — does load-config *evaluate* the globs (scan the filesystem) or just *emit the patterns* for the command to evaluate? The contract lists `prd_globs` as patterns, implying the latter, but the data-model says "detect PRD documents." | Clarify: load-config emits patterns only (deterministic, cheap); the command body does any actual file matching. Pin this so determinism stays clean and the script doesn't do open-ended FS scans. |

### Testing Strategy
| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E6 | 💡 | The suite covers config behaviors well, but the **command body** (harvest, create/amend, semver bump, non-interactive fallback) has no automated test — only manual dogfood. | This is correct per the determinism split (judgment isn't unit-tested), but state it explicitly in the plan so the absence of command-body tests is a documented decision, not an oversight. The quickstart's steps 3-5 are the validation path. |
| E7 | 🎯 | No test asserts invocation from a **different working directory** or that the **bash↔PowerShell parity holds across all four fixtures** (only the default case was manually checked). | Add to the test plan: run each fixture from a non-repo-root CWD, and run the parity assertion over all four fixtures (valid/missing/env/invalid), not just defaults. |

### Operational Readiness
| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E8 | 💡 | CI is specified (Bats on Linux, Pester on Windows) but the plan doesn't say what happens on macOS (the dev's platform), where bash 3.2 differs from Linux bash 5. | Add a macOS bash job (or note bash 3.2 compatibility is covered by the Linux job + explicit 3.2-safe coding). The script already avoids bashisms beyond 3.2; make it a tested guarantee. |

### Dependencies & Integration Risks
| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E9 | 💡 | Reusing core `common.sh` couples load-config to spec-kit internals that could change across `specify` versions. | Acceptable (it's the canonical helper), but the fallback `json_escape` already guards against absence — ensure a test exercises the fallback path (common.sh not sourced). |
