# Critique: vibe-hero MVP (spec 001) — pre-implementation dual-lens review

**Date**: 2026-06-29 · **Reviewer**: main thread (two adversarial subagents were dispatched but hit a tool-environment failure — one returned no findings, the other partially hallucinated spec content; their invented quotes are discarded. This review is grounded in the actual artifacts.)

## Executive Summary

The spec and plan are internally consistent, well-bounded, and technically de-risked (sampling-unavailability, hook correlation, Elo defaults all verified). The **engineering** design is sound with a few real must-fix gaps (concurrent profile writes, judge-gaming, item-difficulty corruption, hook PII). The **product** risk is higher and more strategic: the central bet — that developers want their coding tool to quiz them — is unvalidated, and the MVP is arguably gold-plated relative to that unproven hypothesis. Verdict: **PROCEED WITH UPDATES** — none of the must-address items require re-architecting, but several should be reflected in spec/plan before `/speckit.tasks`.

## Product Lens Findings

**P1 — 🎯 Problem validation.** The core premise (users *want* end-of-work quizzes from their coding tool) has no cited evidence. Mitigation already in spec: quizzes are opt-in offers, never forced — so the downside of being wrong is low (a declined offer). But success should be gated on real accept-rate. *Suggestion:* add a success criterion that tracks **offer accept-rate** and treat it as the go/no-go signal for investing further in content breadth.

**P2 — 💡 MVP scope / gold-plating.** Elo + hysteresis + decay + 5 tiers + free-form handshake + GitHub-fetch + offline fallback is a lot for a v1 whose core hypothesis is unproven; decay constants in particular can't be calibrated against zero data. *Counter-point (why I partially reject this):* the user explicitly chose "depth over breadth — full loop, one tool" and Elo/hysteresis are cheap pure functions, not subsystems. *Suggestion (accepted compromise):* keep Elo (it's the mechanism being proven) but mark **decay/lapse (FR-009/010)** and **free-form judging (US-4)** as the first things to defer if scope pressure appears — both are already isolated behind interfaces. Treat tuning constants as provisional, not validated.

**P3 — 💡 First-run hard gate.** A hard gate (FR-032) is a first-impression cost before any value. Mitigation in design: it only gates *vibe-hero* actions, not the user's actual coding work (the MCP returns SETUP_REQUIRED only when a vibe-hero tool is called) — so it is *not* a tollbooth on real work. *Suggestion:* make this explicit in the spec to avoid the misread — clarify FR-032 gates "any vibe-hero action," NOT the host session or the user's task.

**P4 — 🎯 Decline anti-nag is under-specified.** FR-020 says a decline suppresses offers "for the remainder of the session," but there's no **cross-session** backoff — a user who declines every session still gets offered every session. *Suggestion:* add exponential/again-later backoff across sessions and a global mute after N consecutive declines; make it a measurable criterion.

**P5 — 💡 Outcome vs mechanism.** Quiz-primary scoring measures *knowledge*, but the product's real goal is *better tool usage*. The success criteria measure the mechanism working (SC-009) and learning state, not behavior change. *Suggestion:* add a (later-phase, non-blocking) outcome signal — did the user actually use a taught capability after graduating its topic? — as a north-star, without making usage *score* (preserving FR-005).

## Engineering Lens Findings

**E1 — 🎯 Concurrent profile writes (race condition).** The profile is a single global JSON at `~/.vibe-hero/profile.json`, but a user can run **multiple Claude Code sessions simultaneously**, each with its own MCP server process writing the same file. Last-write-wins will silently lose ability/graduation updates. *Suggestion:* add an atomic write strategy — write-to-temp + rename, plus an advisory lock (e.g. `proper-lockfile`) or per-write read-modify-write under a lock; document it in plan.md. Consider an append-only event log for quiz results as the durable source, with the profile as a rebuildable projection.

**E2 — 🎯 Free-form judge gaming / reliability.** The host agent both *takes* and *judges* the free-form answer (it has the user's answer, the rubric, AND the reference). A lazy or sycophantic model can just return `pass: true`. There's no adversarial check. *Suggestion:* (a) the steering must instruct strict rubric-based judging and require the verdict to cite which criteria were met/missed; (b) keep free-form OUT of the v1 graduation path (it's already deterministic-only for v1 per OD-002) so gaming can't affect real graduation until a verification approach exists; (c) consider requiring the model to output per-criterion booleans, not a single pass.

**E3 — 🎯 Item-difficulty corruption from a single learner.** Research §OD-005 says items "also carry Elo and update." With one learner, letting item difficulty drift from that learner's answers conflates "hard item" with "this user got it wrong," corrupting the difficulty scale and the ability estimate that depends on it. *Suggestion:* for v1, treat **item difficulty as FIXED** (authored difficulty tags only); do not online-update item difficulty from a single learner. Revisit pooled difficulty calibration only when multi-user telemetry exists.

**E4 — 🎯 Hook payload PII / secrets.** PostToolUse payloads include `tool_input`/`tool_output` — which can contain secrets, file contents, prompts. The privacy requirement (FR-018) says "derived signals only," but the *extraction* boundary must be airtight and tested. *Suggestion:* the observation layer must extract `{tool_name, topicKeys, success, timestamp, tool_use_id}` and **never** write `tool_input`/`tool_output` anywhere — add an explicit unit test asserting no raw payload field is persisted, and document the redaction boundary in plan.md.

**E5 — 💡 Timestamp source for decay.** Decay/staleness needs wall-clock time. Fine in the MCP server (Node has `Date.now()`), but note: tests must inject a clock (don't call `Date.now()` directly in the engine) so decay/lapse is deterministically testable. *Suggestion:* pass a `now()` into the pure engine functions; never read the clock inside `engine/`.

**E6 — 💡 Schema versioning / migration.** Both `Profile.schemaVersion` and `CatalogManifest.version` exist, but no migration policy is defined. A fetched catalog newer than the engine's schema, or an old profile, needs a defined behavior. *Suggestion:* define forward/back-compat rules — reject-with-clear-error on unknown major content schema; auto-migrate profile on minor bumps; document in plan.md.

**E7 — 💡 Observation testability + Stop-hook semantics.** Hooks are inherently hard to test. The `ObservationSource` interface helps, but the Stop-hook→`get_offer`→surface path crosses process boundaries. *Suggestion:* make the hook a thin shell that only calls `get_offer` and prints; put ALL logic in the MCP (testable). Already mostly the plan — make it explicit and add an integration test that feeds a synthetic hook payload.

**E8 — 🤔 `@modelcontextprotocol/sdk` + content trust.** Fetched content is data (quiz items), not executed code, so RCE risk is low — but malformed/malicious fetched YAML is still parsed. *Suggestion:* the same Zod validation that guards bundled content MUST run on fetched content before it touches the cache; a failed validation falls back to cache/bundled (already FR-027) — assert this with a test feeding malformed remote content.

## Cross-Lens Insights

- **X1 — 🎯 (P2 × E3 × E2):** The three things most worth de-risking by *deferral* — online item-difficulty updating (E3), free-form judging in the graduation path (E2), and decay tuning (P2) — are all already isolated behind interfaces. Lock them out of the v1 graduation-affecting path: fixed item difficulty, deterministic-only graduation, decay present but flagged provisional. This simultaneously cuts product scope risk and engineering correctness risk.
- **X2 — 💡 (P4 × E7):** The anti-nag backoff (P4) and the offer/Stop-hook plumbing (E7) are the same code path; specify the cadence/backoff state machine once, in the MCP, and test it there.

## Verdict: ⚠️ PROCEED WITH UPDATES

No finding requires re-architecting. The must-address items are: a small set of spec clarifications (P3, P4) and engineering constraints to bake into the plan/tasks (E1–E4, X1). All are additive and most are already partially anticipated by the interface-isolated design.
