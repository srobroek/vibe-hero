# Critique: vibe-hero Distribution (spec 002) — pre-tasks dual-lens review

**Date**: 2026-06-29 · **Reviewer**: main thread (direct; subagents unreliable in this env). Grounded in the real spec/plan/research + live inspection of agentic-packages (origin/main) and the installed apm-cli marketplace client.

## Executive Summary

The plan is well-researched and the security posture (OIDC Trusted Publishers, no long-lived token, atomic publish-before-marketplace) is strong. The npx-launch + single-bin design is sound and reuses existing code. Verdict: **PROCEED WITH UPDATES** — one must-fix is a factual error in the cross-publish mechanism (OD-004), the rest are recommendations that harden release/runtime correctness. Nothing requires re-architecting.

## Product Lens Findings

**P1 — 💡 Cost/benefit of cross-publishing to agentic-packages (3c).** Cross-listing in a second marketplace doubles the release surface for marginal reach (it's the same author's marketplace). Until there's demand, it's speculative scope. *Suggestion:* make cross-publish a **fast-follow / optional** (its own user story flagged P3), ship the vibe-hero root marketplace first; don't block v1 distribution on it.

**P2 — 💡 First-run latency UX (3d).** `npx -y @vibe-hero/server` on first invocation downloads the package — the user's first MCP call (and first Stop-hook offer) pays a multi-second cold fetch with no feedback. *Suggestion:* document the cold-start expectation; consider the setup skill warming the cache (an explicit first `npx` call during setup) so the latency lands during onboarding, not mid-work.

**P3 — 🤔 npx availability assumption (3d/4g).** The whole launch model assumes `npx`/network is available wherever Claude Code runs. Offline-first users or locked-down environments can't fetch the package on first use (bundled *content* is offline, but the *server binary* still needs one network fetch ever). *Suggestion:* state this honestly in install docs; offer a documented "pre-install `@vibe-hero/server` globally" escape hatch for air-gapped setups.

**P4 — 💡 Rollback story (3e).** Floating `latest` means a bad publish reaches every installed user on their next npx resolve, with no pin to roll back to. *Suggestion:* define a rollback procedure (npm `deprecate` + publish a fixed patch; or `dist-tag` the last-good as `latest`). Pairs with E4.

## Engineering Lens Findings

**E1 — 🎯 Cross-publish mechanism is factually wrong (4a/4g).** Research OD-004 says cross-list in agentic-packages via a marketplace `source` "referencing the vibe-hero repo." **Verified false**: in an APM marketplace.json, plugin `source` entries are LOCAL relative paths to packages *inside that same repo* (all agentic-packages entries are `./packages/<name>`). A *marketplace* can be a remote `owner/repo`, but a plugin within it cannot point at a different repo. So "by reference, no duplication" isn't achievable via `source`. *Suggestion:* pick a real mechanism — (a) users add vibe-hero's OWN marketplace (`apm marketplace add srobroek/vibe-hero`) — simplest, no agentic-packages change, recommended; or (b) agentic-packages vendors a thin `mcp-vibe-hero` package (a copy that just declares the npx `.mcp.json`) — accepts controlled duplication. Rewrite OD-004/FR-009/SC-010 to one of these; drop "no duplication AND in agentic-packages by pure reference" as it's not possible.

**E2 — 🎯 `npx -y` without a version pin is a supply-chain + reproducibility risk (4c).** Floating `latest` (FR-012) means every server launch resolves+executes whatever is newest — a compromised or accidentally-broken publish runs immediately on all users, and provenance is verified at publish, not at npx-run. *Suggestion:* this was a deliberate UX call, but at least (a) document the risk, (b) consider `npx -y @vibe-hero/server@^1` (major pin) as a middle ground that still auto-updates within a major — bounds blast radius without losing auto-update. Flag explicitly that floating-latest trades reproducibility/supply-chain-tightness for simplicity (SC-007 already softened to "transient skew").

**E3 — 🎯 Stop hook running `npx` on every turn-end (4b/4d).** Wiring the Stop hook to `npx -y @vibe-hero/server get-offer` means a process spawn + npx resolution at the END OF EVERY agent response. Even cached, that's non-trivial latency on a hot path, and a cold/slow npx could hang the hook. *Suggestion:* (a) keep the hook's existing hard timeout + silent-skip guards (already present); (b) strongly prefer resolving an already-installed binary over `-y` re-fetch; (c) consider the hook calling a long-lived path or the already-running MCP server rather than a fresh npx per stop. At minimum, bound the hook's npx call with an aggressive timeout so it never delays the user.

**E4 — 💡 Atomic publish ordering needs a concrete failure recipe (4f).** FR-016 says "publish before marketplace, fail atomically" but npm publish is not transactional with a git commit. If publish succeeds and the marketplace-commit step fails, you have a published version the marketplace doesn't list (or vice-versa if ordering flips). *Suggestion:* define the exact recovery: publish npm FIRST; only on success regenerate+commit marketplace; if the commit fails, the next release reconciles (idempotent regenerate). Document that npm is the source of truth and the marketplace is a derived pointer.

**E5 — 💡 release-please for a non-monorepo single package (4a).** agentic-packages uses release-please for ~100 packages with tag pattern `{name}-v{version}`. For one package, that config is heavier than needed and the tag pattern is monorepo-shaped. *Suggestion:* use release-please in **single-package mode** (simple `v{version}` tags, one component) — simpler config, same OIDC publish trigger. Note this in the plan so tasks don't copy the monorepo config wholesale.

**E6 — 💡 `prepublishOnly` + provenance build determinism (4f).** Provenance attestation binds the published artifact to the CI build. If `prepublishOnly` rebuilds on the developer's machine during a manual bootstrap publish, the bootstrap won't have provenance (only CI publishes do). *Suggestion:* accept that the bootstrap publish (FR-014a) has no provenance (it's pre-OIDC); all CI publishes do. Document it so it's not mistaken for a regression.

**E7 — 🤔 Bundled-content size + npm package weight (4d).** Copying all of `content/` into the package is fine now (4 small topics) but as the curriculum grows the npm package bloats, and content updates force a server republish (the runtime fetch was supposed to decouple them). *Suggestion:* keep the bundled snapshot deliberately MINIMAL (a baseline subset for offline), and rely on `VIBE_HERO_CONTENT_URL` fetch for the full/updated catalog — so content growth doesn't bloat or churn the npm package.

**E8 — 💡 Test the published artifact, not just the source (4e).** All 144 tests run against `src`. The packaging risks (bin dispatch, files-included, bundled-content-in-dist) only manifest in the BUILT/PACKED artifact. *Suggestion:* add a test that runs against `npm pack` output / the built `dist` (the quickstart V1/V2/V3 already describe this — make them automated CI checks, not manual).

## Cross-Lens Insights

- **X1 — 🎯 (P1 × E1):** Cross-publish is both speculative scope (P1) AND mechanically not what the plan claims (E1). Resolve together: ship vibe-hero's own marketplace first; make agentic-packages cross-listing an explicit later decision with a real mechanism (vendored thin package), not "by reference."
- **X2 — 💡 (P2/P4 × E2/E3):** Floating-latest + npx-per-stop create the cold-start (P2), rollback (P4), supply-chain (E2), and hot-path-latency (E3) concerns — all rooted in the same "resolve latest via npx at runtime" decision. A major-version pin (`@^1`) + a warmed cache + an aggressive hook timeout collectively de-risk all four without abandoning the npx model.
